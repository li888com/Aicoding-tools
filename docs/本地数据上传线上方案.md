## 1. 当前状态分析

### 1.1 本地 → 线上 架构

```
┌──────────────────────────┐       ┌──────────────────────────────┐
│  本地 (MCP Toolbox)        │       │  线上 (ai-test.sbtjt.com)       │
│                          │       │                              │
│  .mcp-toolbox/data.json  │  →→→  │  /api/ai-coding/rounds        │
│  ├─ rounds[]             │  POST  │  /api/ai-coding/requirements  │
│  ├─ roundReverts[]       │  PUT   │  /api/ai-coding/round-reverts │
│  ├─ requirements[]       │  POST  │  /api/ai-coding/token-usage-  │
│  ├─ tokenUsageEvents[]   │  POST  │    events                    │
│  └─ conversations[]      │       │                              │
└──────────────────────────┘       └──────────────────────────────┘
```

### 1.2 ID 差异

| 项目 | 本地 | 线上 |
|------|------|------|
| ID 生成方式 | 本地自增（1, 2, 3...） | 服务端雪花ID（1923847651928374） |
| round.id | 本地自增 | 线上生成，返回给客户端 |
| requirement.requirementId | 用户指定（#44） | 用户指定（保持一致） |

### 1.3 关键问题

1. **ID 映射**：本地的 `round.id=1` 上传后，线上返回 `id=1923847651928374`。后续的 revert 和 token event 需要引用线上 ID
2. **去重**：同一轮不能重复上传，需要幂等机制
3. **增量同步**：每次只上传未同步的记录，避免全量重传
4. **上传顺序**：requirement → round → revert → token event（后三者依赖前者）

## 2. 整体方案

### 2.1 方案选型

采用 **同步脚本 + 本地追踪** 方案：

- 在本地 `data.json` 中为每条记录增加 `_sync` 字段，追踪上传状态
- 编写同步脚本 `scripts/sync-to-online.ts`，读取本地数据，按顺序调用线上 API
- 单向同步：本地 → 线上，线上为数据汇总端

### 2.2 为什么不直接在 MCP tool 里双写？

| 方案 | 优点 | 缺点 |
|------|------|------|
| MCP tool 双写 | 实时 | MCP server 需网络、耦合重、失败影响本地记录 |
| **同步脚本（推荐）** | 解耦、重试友好、批量高效 | 有延迟 |

推荐同步脚本，原因：
- 本地离线也能正常工作（核心流程不受网络影响）
- 上传失败可以重试，不影响本地使用
- 批量上传效率更高

## 3. 数据上传顺序

上传必须严格按以下顺序执行：

```
第1步: requirements
  ↓  线上返回无新ID（requirementId 由用户指定，不变）
第2步: rounds
  ↓  线上返回新 roundId（雪花ID），本地记录映射
第3步: roundReverts
  ↓  targetRoundId 使用步骤2映射后的线上 roundId
第4步: tokenUsageEvents
     roundId 使用步骤2映射后的线上 roundId
```

> 每一步成功后立即更新本地 `_sync` 状态，确保断点续传。

## 4. 实现步骤

### 第1步：扩展本地数据模型

在 `src/local-storage.ts` 的类型定义中增加同步状态字段：

```ts
type SyncStatus = "pending" | "synced" | "skipped" | "failed";

// 为 Round、RoundRevert、TokenUsageEvent 增加：
type Round = {
  // ... 原有字段
  _sync?: {
    status: SyncStatus;
    onlineId?: number;      // 线上返回的id
    syncedAt?: string;      // 同步时间
    error?: string;         // 失败原因
  };
};
```

**向前兼容**：`_sync` 为可选字段，旧数据没有该字段时视为 `pending`。

### 第2步：编写同步脚本

创建 `scripts/sync-to-online.ts`，核心逻辑：

```
1. 读取 data.json
2. 筛选 _sync.status !== "synced" 的记录
3. 按顺序上传：
   - syncRequirements()
   - syncRounds()          → 返回 onlineId，存入本地映射
   - syncRoundReverts()    → targetRoundId 替换为 onlineId
   - syncTokenUsageEvents() → roundId 替换为 onlineId
4. 每次成功后立即保存 data.json（断点续传）
5. 输出同步报告
```

### 第3步：配置线上地址和认证

通过环境变量配置：

```bash
# .env
SYNC_API_BASE_URL=https://ai-test.sbtjt.com/api/ai-coding
SYNC_API_TOKEN=<线上API的Bearer Token>
```

### 第4步：添加 npm script

```json
{
  "scripts": {
    "sync:online": "tsx scripts/sync-to-online.ts",
    "sync:online:dry": "tsx scripts/sync-to-online.ts --dry-run"
  }
}
```

- `sync:online`：执行实际上传
- `sync:online:dry`：模拟运行，只打印会上传什么，不实际调用 API

## 5. 详细设计

### 5.1 API 调用映射

#### requirements 上传

```
本地数据                    线上 API
──────────────────────────────────────────────
requirementId: 44    →    PUT /api/ai-coding/requirements/44
title: "xxx"               Body: { title, projectName, gpmNumber, status }
projectName: "mcp"
gpmNumber: null
status: "active"
```

调用后无新ID，`requirementId` 不变。

#### rounds 上传

```
本地数据                    线上 API
──────────────────────────────────────────────
id: 1                 →    POST /api/ai-coding/rounds
conversationId: "xxx"      Body: {
startedAt: "..."             idempotencyKey: "local-round-1",  ← 用本地ID作幂等键
endedAt: "..."               conversationId, startedAt, endedAt,
modelName: "..."             modelName, promptText, requirementId,
requirementId: 44            filesChanged, linesAdded, linesDeleted,
...                          inputTokens, outputTokens, totalTokens,
                             metadata: { ..., localRoundId: 1 }
                           }
                           
                           ←  返回 { id: 1923847651928374, ... }
                           
                           记录映射: localRoundId 1 → onlineRoundId 1923847651928374
```

**关键**：`idempotencyKey` 使用 `"local-round-{本地id}"` 格式，确保同一轮不会因重试而重复创建。

#### roundReverts 上传

```
本地数据                    线上 API
──────────────────────────────────────────────
targetRoundId: 1      →    POST /api/ai-coding/round-reverts
                            Body: {
                              targetRoundId: 1923847651928374,  ← 用映射后的线上ID
                              conversationId, revertedAt, modelName,
                              reason, ...
                            }
```

**注意**：`targetRoundId` 必须替换为步骤2中映射得到的线上 roundId。

#### tokenUsageEvents 上传

```
本地数据                    线上 API
──────────────────────────────────────────────
roundId: 1            →    POST /api/ai-coding/token-usage-events
                            Body: {
                              roundId: 1923847651928374,  ← 用映射后的线上ID
                              client, sourcePath, sourceEventId,
                              inputTokens, outputTokens, totalTokens, ...
                            }
```

### 5.2 同步状态机

```
                  ┌──────────┐
                  │ pending  │  (新记录 / _sync字段不存在)
                  └────┬─────┘
                       │ syncRequirements/syncRounds/...
          ┌────────────┼────────────┐
          ↓            ↓            ↓
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ synced   │ │ skipped  │ │ failed   │
    └──────────┘ └──────────┘ └────┬─────┘
        完成        手动跳过       │ 重试
                                   │
                              ┌────┴─────┐
                              │ synced   │ (重试成功)
                              └──────────┘
```

- **pending**：未上传
- **synced**：已上传成功，记录了 `onlineId`
- **skipped**：手动跳过（如该轮次数据不完整，决定不上传）
- **failed**：上传失败，下次运行时会重试

### 5.3 ID 映射表

同步过程中在内存维护一个映射表：

```ts
const idMap = new Map<number, number>();
// localRoundId → onlineRoundId
// 例: 1 → 1923847651928374
```

这个映射表用于后续 revert 和 token event 的 `targetRoundId` / `roundId` 替换。

同步完成后，映射关系保存在各记录的 `_sync.onlineId` 中，下次运行直接从本地读取，无需重复查询线上。

### 5.4 断点续传

每完成一个步骤（或每成功上传一条记录），立即 `saveData()` 保存整个 data.json。

这样即使中途中断，已同步的记录不会重复上传，下次运行从断点继续。

### 6.1 首次使用

```bash
# 1. 配置环境变量
cat >> .env << 'EOF'
SYNC_API_BASE_URL=https://ai-test.sbtjt.com/api/ai-coding

# 2. 试运行（不实际调用API）
npm run sync:online:dry

# 3. 正式上传
npm run sync:online
```

每次 AI Coding 工作结束后运行一次：


## 7. 风险与注意事项

### 7.1 数据一致性

| 场景 | 处理方式 |
|------|----------|
| 上传到一半中断 | 断点续传，下次运行从上次成功的记录之后继续 |
| 同一轮重复上传 | `idempotencyKey` 保证幂等，线上返回已有记录 |
| 线上已存在但本地无 `_sync` 标记 | 手动标记为 `skipped` 或首次运行后自动修正 |
| 本地 round 的 requirementId 对应的需求在线上不存在 | 需先运行 requirement 上传步骤，会自动创建 |
