# AI Coding Stats 线上存储改造方案

这份文档说明如何把当前基于本地 `.mcp-toolbox/data.json` 的存储方式，改造成写入线上接口。目标是让 Claude Code / Codex 等 AI 客户端在每轮工作结束后，把统计数据统一上报到服务端，并支持 Dashboard 从线上数据展示统计结果。

## 1. 改造目标

当前链路：

```text
AI 客户端调用 MCP tool
  -> record_ai_coding_round
  -> src/database.ts
  -> src/local-storage.ts
  -> .mcp-toolbox/data.json
  -> Dashboard 读取本地 JSON 汇总展示
```

目标链路：

```text
AI 客户端调用 MCP tool
  -> record_ai_coding_round
  -> 调用线上 API
  -> 线上数据库
  -> Dashboard 读取线上 API 汇总展示
```

建议先实现最小闭环：

```text
POST /api/ai-coding/rounds
PUT  /api/ai-coding/requirements/:id
GET  /api/ai-coding/dashboard/*
```

撤销接口和 Token 日志补录接口可以第二阶段再做。

## 2. 推荐分阶段

### 第一阶段：先让轮次统计上云

先把 `record_ai_coding_round` 写入本地 JSON 的逻辑，替换成调用线上 `POST /api/ai-coding/rounds`。

第一阶段只要保证这些页面能有数据：

- 总览
- 按需求
- 按模型
- 按日期
- 对话详情

这一阶段暂时可以不做 Token 日志补录证据。

### 第二阶段：把需求标题自动保存

你最关心的是“在 Claude Code 里写标题，然后统计页能看到标题”。所以建议扩展 `record_ai_coding_round` 的入参，增加这些字段：

```json
{
  "requirementTitle": "登录页改造",
  "projectName": "用户中心",
  "gpmNumber": "GPM-12345"
}
```

服务端收到 `requirementId` 和 `requirementTitle` 后，自动 upsert 需求记录。

### 第三阶段：再做撤销和 Token 补录

等核心统计稳定后，再增加：

- `POST /api/ai-coding/round-reverts`
- `POST /api/ai-coding/token-usage-events`

这样可以避免一开始接口过大，调试成本过高。

## 3. 核心接口设计

### 3.1 记录一轮 AI Coding

```http
POST /api/ai-coding/rounds
```

请求体：

```json
{
  "conversationId": "claude:C:/project/demo",
  "startedAt": "2026-05-19T00:10:00.000Z",
  "endedAt": "2026-05-19T00:18:30.000Z",
  "modelName": "claude-3-7-sonnet",
  "promptText": "#12 登录页改造",
  "requirementId": 12,
  "requirementSource": "prompt",
  "requirementTitle": "登录页改造",
  "projectName": "用户中心",
  "gpmNumber": "GPM-12345",
  "filesChanged": 3,
  "linesAdded": 80,
  "linesDeleted": 12,
  "codeLinesChanged": 92,
  "inputTokens": 4200,
  "outputTokens": 1800,
  "totalTokens": 6000,
  "tokenSource": "mcp_payload",
  "metadata": {
    "client": "claude-code",
    "projectPath": "C:/project/demo",
    "threadId": "session-id",
    "turnId": "message-id",
    "codeStatsSource": "git diff --numstat"
  }
}
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `conversationId` | 是 | 稳定会话 ID，用于需求上下文继承 |
| `startedAt` | 是 | 本轮开始时间，ISO 8601 |
| `endedAt` | 是 | 本轮结束时间，ISO 8601 |
| `modelName` | 是 | 本轮使用的模型 |
| `promptText` | 否 | 用户原始需求文本，用于解析 `#12` 和审计 |
| `requirementId` | 否 | 需求编号，可由 MCP 端解析，也可由服务端解析 |
| `requirementSource` | 否 | `prompt` / `context` / `empty` |
| `requirementTitle` | 否 | 需求标题；有值时服务端建议自动 upsert 需求 |
| `projectName` | 否 | 项目名 |
| `gpmNumber` | 否 | GPM 编号 |
| `filesChanged` | 否 | 修改文件数 |
| `linesAdded` | 否 | 新增行数 |
| `linesDeleted` | 否 | 删除行数 |
| `codeLinesChanged` | 否 | 代码改动行数，默认可等于新增+删除 |
| `inputTokens` | 否 | 输入 Token |
| `outputTokens` | 否 | 输出 Token |
| `totalTokens` | 否 | 总 Token |
| `tokenSource` | 否 | `mcp_payload` / `claude_jsonl` / `codex_log` / `unavailable` |
| `metadata` | 否 | 扩展字段，建议保留客户端、项目路径、线程 ID 等 |

推荐返回：

```json
{
  "id": 123,
  "conversationId": "claude:C:/project/demo",
  "requirementId": 12,
  "requirementSource": "prompt",
  "modelName": "claude-3-7-sonnet",
  "durationMs": 510000,
  "codeLinesChanged": 92,
  "totalTokens": 6000,
  "requirement": {
    "requirementId": 12,
    "title": "登录页改造",
    "projectName": "用户中心",
    "gpmNumber": "GPM-12345"
  }
}
```

```http

```

请求体：

```json
{
  "title": "登录页改造",
  "projectName": "用户中心",
  "gpmNumber": "GPM-12345",
  "status": "active",
  "description": "改造登录页 UI 和校验逻辑"
}
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `title` | 否 | 需求标题 |
| `projectName` | 否 | 项目名 |
| `gpmNumber` | 否 | GPM 编号 |
| `status` | 否 | `active` / `done` / `archived` |
| `description` | 否 | 备注或验收说明 |

推荐规则：

- 如果 `POST /rounds` 里带了 `requirementTitle`，服务端自动 upsert requirement。
- 如果后续用户在 Dashboard 手动修改标题，以 `PUT /requirements/:id` 为准。
- 不建议把需求标题完全依赖 `promptText` 自动截取，因为用户输入格式不稳定。

### 3.3 记录撤销事件

第二阶段再做：

```http
POST /api/ai-coding/round-reverts
```

请求体：

```json
{
  "conversationId": "claude:C:/project/demo",
  "targetRoundId": 123,
  "revertedAt": "2026-05-19T00:30:00.000Z",
  "modelName": "claude-3-7-sonnet",
  "promptText": "撤销上一轮代码改动",
  "reason": "user requested undo",
  "filesChanged": 2,
  "linesAdded": 10,
  "linesDeleted": 80,
  "codeLinesChanged": 90,
  "metadata": {
    "client": "claude-code"
  }
}
```

注意：撤销不是删除原 round，而是新增一条 revert 记录。统计“有效轮次”时排除已被 revert 的 target round。

### 3.4 Token 日志补录证据

第二阶段或第三阶段再做：

```http
POST /api/ai-coding/token-usage-events
```

这个接口不是给人手动调用的，通常由日志同步脚本或定时任务调用。作用是：当 `POST /rounds` 时没有拿到准确 Token，后续从 Claude Code / Codex 本地日志中找到 usage 记录，再回填 Token，并保存“这个 Token 数字从哪里来的”。

请求体：

```json
{
  "roundId": 123,
  "client": "claude-code",
  "sourcePath": "C:/Users/xxx/.claude/projects/project.jsonl",
  "sourceEventId": "event-uuid",
  "conversationId": "session-id",
  "turnId": "message-uuid",
  "modelName": "claude-3-7-sonnet",
  "startedAt": "2026-05-19T00:10:20.000Z",
  "endedAt": "2026-05-19T00:10:20.000Z",
  "inputTokens": 4200,
  "outputTokens": 1800,
  "totalTokens": 6000,
  "rawEvent": {}
}
```

建议服务端收到后同时做两件事：

1. 更新对应 `round` 的 Token 字段。
2. 插入一条 `token_usage_events` 证据记录。

## 4. 接口认证与错误码（已实现）

线上接口需要同时服务两类调用方：

- MCP server：用于写入 round、requirement、revert、token usage event。
- Dashboard：用于查询 summary、requirements、models、timeline、rounds 等统计数据。

当前认证与错误码能力按“已实现”处理，后续 MCP 端只需要按这里的约定接入。

### 4.1 认证方式

MCP server 调用线上 API 时，统一使用 Bearer Token：

```http
Authorization: Bearer <AI_CODING_API_TOKEN>
```

MCP 端配置：

```env
AI_CODING_API_BASE_URL=https://example.com
AI_CODING_API_TOKEN=replace-with-token
AI_CODING_API_TIMEOUT_MS=10000
```

服务端处理规则：

- 写接口必须校验 `Authorization`。
- Token 缺失、格式错误或无效时返回 `401 unauthorized`。
- Token 有效但没有对应资源权限时返回 `403 forbidden`。
- 不建议把 token 写入日志；错误日志中只记录请求路径、状态码、requestId。

### 4.2 推荐错误响应结构

所有非 2xx 响应建议统一返回：

```json
{
  "error": "validation_error",
  "message": "startedAt must be an ISO 8601 datetime",
  "requestId": "req_202605190001",
  "details": {
    "field": "startedAt"
  }
}
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `error` | 是 | 稳定错误码，给程序判断 |
| `message` | 是 | 可读错误说明，给人排查 |
| `requestId` | 否 | 请求追踪 ID |
| `details` | 否 | 字段级或业务级详情 |

### 4.3 错误码约定

| HTTP 状态码 | `error` | 场景 |
| --- | --- | --- |
| `400` | `validation_error` | 请求体字段格式错误，例如时间不是 ISO 8601 |
| `400` | `invalid_requirement_status` | `status` 不是 `active` / `done` / `archived` |
| `401` | `unauthorized` | 缺少 token、token 格式错误或 token 无效 |
| `403` | `forbidden` | token 有效，但没有当前项目/租户权限 |
| `404` | `not_found` | 指定的 round、requirement 或 revert 目标不存在 |
| `409` | `duplicate_idempotency_key` | 幂等键冲突，且请求体与历史请求不一致 |
| `409` | `round_already_reverted` | 目标 round 已经被撤销 |
| `422` | `unmatched_round` | Token 补录找不到可更新的 round |
| `429` | `rate_limited` | 调用频率超过限制 |
| `500` | `internal_error` | 服务端未知错误 |
| `503` | `service_unavailable` | 数据库或依赖服务不可用 |

### 4.4 幂等请求的返回规则

`POST /api/ai-coding/rounds` 建议带 `idempotencyKey`：

```json
{
  "idempotencyKey": "claude-code:session-id:message-id"
}
```

处理规则：

- 首次请求：创建 round，返回 `201` 或 `200`。
- 重复请求且 payload 一致：返回已创建的 round，状态码可为 `200`。
- 重复请求但 payload 不一致：返回 `409 duplicate_idempotency_key`。
- 没传 `idempotencyKey` 时，可用 `metadata.client + conversationId + metadata.turnId` 作为兜底唯一键。

## 6. 失败重试与本地兜底

线上化后，MCP server 的写入链路会依赖网络和服务端可用性。建议不要让一次网络抖动直接丢数据。

### 6.1 写入策略

推荐策略：

1. 优先调用线上 API。
2. 如果线上 API 成功，直接返回线上结果。
3. 如果线上 API 失败，按错误类型决定是否重试。
4. 如果重试后仍失败，将原始 payload 写入本地 pending 队列。
5. 后续由同步命令或后台任务重放 pending 队列。

### 6.2 哪些错误可以重试

建议重试：

- 网络超时
- DNS/连接失败
- `429 rate_limited`
- `500 internal_error`
- `503 service_unavailable`

不建议重试：

- `400 validation_error`
- `401 unauthorized`
- `403 forbidden`
- `404 not_found`
- `409 duplicate_idempotency_key`

### 6.3 本地 pending 队列

如果线上写入失败但 payload 本身有效，建议保存到：

```text
.mcp-toolbox/pending-remote-rounds.jsonl
```

每行一条 JSON，建议字段：

```json
{
  "idempotencyKey": "claude-code:session-id:message-id",
  "endpoint": "/api/ai-coding/rounds",
  "method": "POST",
  "payload": {},
  "failedAt": "2026-05-19T10:00:00.000Z",
  "attempts": 3,
  "lastError": {
    "error": "service_unavailable",
    "message": "database unavailable"
  }
}
```

### 6.4 重放队列

建议新增命令：

```bash
npm run remote:sync-pending
```

重放规则：

- 按文件顺序重放。
- 成功后从 pending 队列移除。
- 碰到 `400/401/403` 这类不可重试错误，移动到 dead-letter 文件。
- `idempotencyKey` 必须保持不变，避免重放造成重复 round。

建议 dead-letter 文件：

```text
.mcp-toolbox/dead-letter-remote-rounds.jsonl
```

## 7. 数据库 SQL 设计（已确定）

以下 DDL 作为线上存储的确定表结构。字段采用业务系统常见的 `id` 主键、`del_flag` 软删除、`create_time/update_time` 审计字段。

说明：

- `id` 未设置自增，默认由业务侧生成（例如雪花 ID 或统一 ID 服务）。
- `ai_coding_rounds` 已包含 `client/project_path/thread_id/turn_id/code_stats_source` 等常用 metadata 字段；完整原始扩展信息仍可放入 `metadata`。
- 当前 SQL 未单独设计 `ai_coding_conversations` 表。若需要严格保留 `conversationId -> currentRequirementId` 上下文，可从同一 `conversation_id` 最近一条有效 round 推导，或后续补充独立 conversation 表。

```sql
CREATE TABLE IF NOT EXISTS `ai_coding_requirements` (
  `id` bigint NOT NULL COMMENT '主键ID',
  `requirement_id` bigint NOT NULL COMMENT '需求编号，MCP侧传入的业务ID',
  `title` varchar(255) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '需求标题',
  `project_name` varchar(255) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '项目名称',
  `gpm_number` varchar(100) COLLATE utf8mb4_bin DEFAULT NULL COMMENT 'GPM编号',
  `status` varchar(32) COLLATE utf8mb4_bin DEFAULT 'active' COMMENT '状态：active-进行中 done-完成 archived-归档',
  `description` text COLLATE utf8mb4_bin COMMENT '描述',
  `del_flag` char(1) COLLATE utf8mb4_bin DEFAULT '0' COMMENT '删除标记',
  `create_time` datetime DEFAULT NULL COMMENT '创建时间',
  `update_time` datetime DEFAULT NULL COMMENT '修改时间',
  `create_by` varchar(255) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '创建人',
  `update_by` varchar(255) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '修改人',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_ai_coding_requirements_requirement_id` (`requirement_id`),
  KEY `idx_ai_coding_requirements_status` (`status`),
  KEY `idx_ai_coding_requirements_gpm_number` (`gpm_number`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_bin COMMENT='AI Coding需求信息表';

CREATE TABLE IF NOT EXISTS `ai_coding_rounds` (
  `id` bigint NOT NULL COMMENT '主键ID',
  `conversation_id` varchar(500) COLLATE utf8mb4_bin NOT NULL COMMENT '会话ID',
  `idempotency_key` varchar(500) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '幂等键',
  `started_at` datetime NOT NULL COMMENT '开始时间',
  `ended_at` datetime NOT NULL COMMENT '结束时间',
  `duration_ms` bigint DEFAULT NULL COMMENT '持续时间毫秒',
  `model_name` varchar(255) COLLATE utf8mb4_bin NOT NULL COMMENT '模型名称',
  `prompt_text` longtext COLLATE utf8mb4_bin COMMENT '用户原始需求文本',
  `requirement_id` bigint DEFAULT NULL COMMENT '需求编号',
  `requirement_source` varchar(32) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '需求来源：prompt/context/empty',
  `client` varchar(64) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '客户端：claude-code/codex',
  `project_path` varchar(1000) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '项目路径',
  `thread_id` varchar(255) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '线程ID',
  `turn_id` varchar(255) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '轮次消息ID',
  `code_stats_source` varchar(255) COLLATE utf8mb4_bin DEFAULT NULL COMMENT '代码统计来源',
  `files_changed` int DEFAULT 0 COMMENT '修改文件数',
  `lines_added` int DEFAULT 0 COMMENT '新增行数',
  `lines_deleted` int DEFAULT 0 COMMENT '删除行数',
  `code_lines_changed` int DEFAULT 0 COMMENT '代码改动行数',
```

## 8. MCP 端需要怎么改

建议新增一个线上存储实现，而不是直接删除本地存储。

推荐做法：

```text
src/database.ts
  -> 根据 AI_CODING_STORAGE_MODE 判断用哪种存储

AI_CODING_STORAGE_MODE=local
  -> 当前 local-storage.ts

AI_CODING_STORAGE_MODE=remote
  -> 新增 remote-storage.ts，调用线上 API
```

新增环境变量：

```env
AI_CODING_STORAGE_MODE=remote
AI_CODING_API_BASE_URL=https://example.com
AI_CODING_API_TOKEN=replace-with-token
AI_CODING_API_TIMEOUT_MS=10000
```

这样保留本地模式作为兜底，线上模式逐步切换，风险更低。

## 9. 服务端处理规则

### 9.1 requirementId 解析放哪边

推荐：MCP 端和服务端都支持解析。

- MCP 端：继续按当前逻辑从 `promptText` 解析 `#12`
- 服务端：如果请求没传 `requirementId`，也尝试从 `promptText` 解析

这样即使某些客户端漏传 `requirementId`，服务端也能兜底。

### 9.2 conversation 上下文继承

当前 SQL 未单独设计 conversation 表，因此服务端可以通过 `ai_coding_rounds` 推导 `conversationId -> currentRequirementId`。

规则：

1. 当前请求带 `requirementId`：本轮直接使用该需求编号，并写入 `ai_coding_rounds.requirement_id`
2. 当前请求没带 `requirementId`：查询同一 `conversation_id` 下最近一条未删除、未撤销且 `requirement_id` 不为空的 round，沿用它的需求编号
3. 两者都没有：本轮 `requirementId=null`

推荐查询条件：

```sql
SELECT requirement_id
FROM ai_coding_rounds
WHERE conversation_id = ?
  AND requirement_id IS NOT NULL
  AND del_flag = '0'
  AND is_reverted = 0
ORDER BY ended_at DESC, create_time DESC
LIMIT 1;
```

### 9.3 幂等性

建议 `POST /rounds` 支持幂等键，避免 MCP client 重试导致重复写入。

可选字段：

```json
{
  "idempotencyKey": "claude-code:session-id:message-id"
}
```

服务端可以对 `idempotencyKey` 建唯一索引。

如果没有显式传 `idempotencyKey`，可以用下面字段组合兜底：

```text
client + conversationId + turnId
```

## 10. Dashboard 改造建议

- 本项目 MCP server 只负责采集与上报
